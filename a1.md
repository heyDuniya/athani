## 2. Hex in one page

Some of the team is new to hex. Skip this if you aren't.

Cockburn's insight (2005): **the business logic must not depend on the technologies that surround it.** Dependencies flow inward. Technologies depend on business logic; business logic depends on nothing.

```
        INBOUND ADAPTERS (REST, Kafka in, ASA socket in, WS)
                        │ calls inbound ports
                        ▼
        APPLICATION (use cases) ── defines ports
        DOMAIN (pure business)   ── invariants, state machines
                        │ calls outbound ports
                        ▼
        OUTBOUND ADAPTERS (Postgres, Kafka out, ASA out, ticketing)
```

**Vocabulary.**
- **Domain** — financial concepts (`Order`, `Fill`, `Execution`). Pure Java. No Spring, no JPA, no Jackson.
- **Application** — use cases that orchestrate domain operations. Defines ports (interfaces).
- **Port** — interface describing what the application offers (inbound) or needs (outbound).
- **Adapter** — concrete implementation bridging a port to a real technology.

**The one rule that matters: dependencies point inward.** Adapters → application → domain → nothing. Domain and application contain no framework imports. That rule is what we are debating how to enforce.

## 3. What goes in domain vs application

This is the question that comes up most once code starts being written, and getting it wrong is the most common reason hex implementations fail. Both layers contain business logic — different kinds. Worth treating with care because vague rules here lead to anaemic domain (logic drains into use cases), or to bleeding (use cases re-implement domain rules and drift).

There is no single test that perfectly classifies every rule. Multiple lenses exist; they mostly agree, and the cases where they disagree are usually the architecturally interesting ones. Best to know all of them.

### 3.1 Five lenses for the same question

**Lens 1 — External dependencies.** *Does this rule need infrastructure to evaluate?*
If the rule needs a database, a port, an event publisher, a transaction, an ID generator, the wall clock as a dependency, or coordination across aggregates → **application**. If it can be evaluated purely from values you already hold → **domain**.

**Lens 2 — Principle vs policy.** *Is this intrinsic to what the concept is, or an organisational choice that could change?*
- **Principles** — true by definition of the concept. An order has positive quantity. A fill belongs to exactly one order. Cumulative fills cannot exceed order quantity. These do not change because management changes their mind. → **Domain.**
- **Policies** — organisational choices that could be different next year. Orders over $5M need supervisor approval. Manual fills must be priced within 5% of last trade. The supervisor threshold is $10M for fixed income but $5M for equities. → **Application** (often parameterised, with the parameter loaded via a config port).

A good test: if a regulator forced us to change this rule next quarter, would the *meaning of an order* change, or just the *workflow around handling it*? If just the workflow → application.

**Lens 3 — Whiteboard test.** *Would a trader, sketching the business on a whiteboard with no computers in the room, recognise this rule?*
"A buy order matched with a sell at an agreed price produces a fill" → yes, trader nods → **domain**.
"After a fill is recorded, we publish an event so the ticketing service knows to start working" → no, trader doesn't think in events → **application**.

**Lens 4 — Aggregate boundary.** *Does the rule operate within one aggregate, across multiple aggregates, or against external systems?*
- Within one aggregate (`Execution.applyFill(...)` checks against `Execution`'s own fills) → **domain on the aggregate**.
- Across multiple aggregates that can be loaded together as a coherent unit → **domain service** (still domain — see §3.4).
- Across systems / requires fetching data from outside → **application**.

**Lens 5 — Stability under technical change.** *Would this rule change if we replaced our database, our messaging system, or our wire protocol?*
If yes → it's coupled to technology and belongs in **adapter** (or the rule is wrong).
If no → it's either domain (intrinsic business meaning) or application (orchestration that happens to call ports — but the orchestration shape is stable even if individual ports are reimplemented).

### 3.2 Quick reference

| Domain (rich, intrinsic) | Application (orchestration) |
|---|---|
| Invariants on construction (`Order` qty > 0, `Fill` qty ≤ open qty) | Loading aggregates from repositories |
| State machine transitions (`WORKING → FILLED` legal, `CANCELLED → FILLED` not) | Authorization checks against external systems |
| Self-consistency (`Execution.applyFill(...)` rejects over-fills) | Sequencing operations (load → mutate → persist → publish) |
| Pure calculations (weighted average price, allocation amounts) | Transaction boundaries |
| Value objects with self-validation (`Isin`, `Money`) | Idempotency on retry (deduping by external IDs) |
| Domain events as types (`FillApplied`, `OrderRejected`) | Publishing events through outbound ports |
| Algorithms that operate on aggregate state | Coordinating across aggregates that need separate loads |
| Domain services for cross-aggregate domain rules (§3.4) | Policy-driven workflow (approval thresholds, business hours, halt rules) |

### 3.3 A worked example — `RecordManualFillUseCase`

```
1. authorizationPort.checkTraderCanFill(trader, desk)        ← APPLICATION (port)
2. if not authorized → throw                                  ← APPLICATION
3. execution = executionRepository.findById(executionId)     ← APPLICATION (port)
4. fill = Fill.from(instruction)                              ← DOMAIN (constructor validates qty > 0, price > 0, currency match)
5. execution.applyFill(fill)                                  ← DOMAIN (invariant: cumulative ≤ order qty; state transition)
6. executionRepository.save(execution)                        ← APPLICATION (port)
7. eventPublisher.publish(execution.uncommittedEvents())     ← APPLICATION (port)
```

Steps 4 and 5 are domain. Everything else is application orchestration. The domain doesn't know about repositories, publishers, or authorization. The application doesn't know what makes a fill valid — it just asks the domain to do the work.

**Slogan: rich domain, thin application.**

### 3.4 The cases that confuse everyone

These are the genuinely tricky ones. They show up in every OMS and deserve direct treatment.

**Compliance and limit checks.**
"This trader cannot exceed $10M intraday exposure." This *feels* like business logic — and it is — but evaluating it requires loading other orders, querying a position service, and applying a current-day clock. Pattern:
- **Application** orchestrates: fetches the trader's open orders (port), fetches current position (port), passes them to a domain function.
- **Domain** decides the verdict: a function `LimitCheck.evaluate(orders, position, limit) → ComplianceVerdict` is pure and lives in the domain. The verdict shape (`Verdict.APPROVED`, `Verdict.REJECTED(reason)`) is a domain value.

The principle: **the application gathers the inputs; the domain decides the answer.** Compliance verdicts are domain concepts even though gathering the inputs requires infrastructure.

**Pricing or calculation that needs market data.**
"Mark this order to current bid." The arithmetic is domain. The market-data lookup is application. Same pattern as above: application fetches, passes to a domain function (`Order.markToMarket(price)`), receives a domain result.

**Workflow rules — "approval needed if over X."**
"Orders over $5M need supervisor approval before routing." This looks domain-flavoured but is actually a *policy*: the threshold can change, the approver list can change, the rule could be removed entirely without changing what an order *is*. → **Application**, ideally with the threshold loaded via a config port. The domain still owns what an `ApprovalDecision` *is*, but the workflow logic ("if needs approval, route to inbox; on approval, route to execution") is application.

**Cross-aggregate invariants.**
"Total exposure across all of a trader's open orders cannot exceed limit." No single aggregate owns all open orders, so the rule cannot live on an aggregate. Two valid placements:
- **Domain service** — a class in `*-domain` (e.g., `TraderExposureCheck`) that takes the relevant aggregates as parameters and returns a verdict. Pure Java, no ports. The application loads the orders and calls the domain service. This is the DDD-orthodox answer and the right one for genuine domain rules across aggregates.
- **Application orchestration** — only if the rule is more about workflow than business meaning.

**Idempotency.**
Almost always **application**. "If we've already processed this `ExecutionReport.id`, don't apply it again" is a distributed-system concern, not a business rule. The domain doesn't know that messages can be redelivered.

**Time and the clock.**
A subtle one. "An order expires at end of day" is **domain** (the rule is intrinsic), but reading "what time is it now" is **application** (or an injected `Clock` port). Domain functions should take time as a parameter when they need it; they shouldn't read the system clock themselves. This makes them deterministic and testable.

**Logging, metrics, tracing.**
Always **adapter** (or instrumentation around adapters). Never domain. If a domain method needs to log, it's a sign that what it actually wants is to *return* a richer result that the application can act on (and log).

### 3.5 Two failure modes to avoid

**1. Anaemic domain — the most common hex failure.**
Symptoms: aggregates are bags of getters and setters; all logic lives in use cases; use cases are 200 lines long; the same domain check is repeated in three different use cases.

This happens when developers default to "Spring Boot with hex labels." The compiler can't help them — they have to choose discipline. Under packages-only this is the *default outcome*, because there's no friction stopping logic from drifting into the use case.

If you find yourself writing `if (execution.getState() == FILLED && execution.getQuantity() ...)` in a use case, stop. The decision belongs in `Execution.canAcceptFill(...)` — move it.

**2. Domain bleeding into application.**
Symptoms: use cases reach into aggregates via setters and mutate internals; the same invariant is enforced in multiple places and drifts.

The fix: aggregates expose intent-revealing methods (`Execution.applyFill(...)`, `Execution.cancel(...)`), not setters. Use cases call methods, never reach in.

### 3.6 The connection back to the submodule decision

Section 3 reads like a separate concern, but it isn't. **Anaemic domain is the default failure mode under packages-only**, because there's no compile-time pressure stopping a developer from writing domain logic in the use case. Submodules create that pressure: when a use case needs to do domain-shaped work, the developer either (a) puts it on the aggregate, where it belongs, or (b) fights the build system. (b) is annoying, so (a) wins.

This is one of the strongest practical arguments for submodules — and it's a reason the "we'll just have discipline about packages" position predictably fails over a 10-year horizon.

## 4. Decision 1 — Submodules vs Packages

Two proposals on the table:

- **Proposal A (mine):** each service is a Maven parent module with four submodules (`domain`, `application`, `adapter`, `service`). Dependency direction declared in `pom.xml`, enforced by the compiler.
- **Proposal B (some colleagues):** one Maven module per service. Hexagonal layers as Java packages. Direction enforced by convention, optionally ArchUnit.

### 4.1 The acid test

Most of this debate becomes simple once we ask one question.

Every well-organised Spring Boot application has separate packages for different concerns:

| Standard Spring Boot | "Packages-only hex" (just renamed) |
|---|---|
| `controller/` | `adapter/inbound/rest/` |
| `service/` | `application/usecase/` |
| `repository/` | `adapter/outbound/persistence/` |
| `entity/` | `adapter/outbound/persistence/entity/` |

If "hexagonal with packages only" just means renaming the right column to the left, what have we structurally changed? The `@SpringBootApplication` still pulls everything together. Every class shares the same classpath. `@Autowired` works everywhere. `@Entity` can appear anywhere. A class in `application/usecase/` can directly import a JPA entity from `adapter/outbound/persistence/entity/` — the compiler has no opinion.

**Hexagonal is not a naming convention. It is a dependency structure.**

The acid test is one question:

> **Can the domain code compile without Spring, JPA, Jackson, or any other framework on its classpath?**

| | Standard Spring Boot | Packages-only "hex" | Submodule hex |
|---|---|---|---|
| Spring on domain classpath | Yes | **Yes** | **No** |
| Can `@Autowired` in domain | Yes | **Yes** | **No** |
| Can `@Entity` in domain | Yes | **Yes** | **No** |
| Direction enforced | No | No (ArchUnit at test time) | **Compile-time** |
| Framework-free domain tests | No | No | **Yes** |
| What it actually is | Layered Spring Boot | Layered Spring Boot, renamed | **Hexagonal** |

On every row that matters, packages-only is identical to standard Spring Boot. Only the folder names differ.

By the packages-only logic, every well-organised Spring Boot app in the world is hexagonal — because they all have controllers, services, and repositories in separate packages. That is obviously not what hex means.

**The implied choice.** If we go packages-only, the honest framing is: *"We are writing a well-layered Spring Boot application."* That is a legitimate choice — but it cannot be marketed as hexagonal, and we have to drop the architectural promises (framework-free testing, technology replaceability, domain longevity) that motivated the hex conversation in the first place.

We pick one of:
1. **"We are doing hex"** → submodules. That is what doing hex actually looks like.
2. **"We are doing layered Spring Boot"** → fine, but the pitch to leadership changes, and we need a different story for why Mercury will avoid C3+'s fate.

We cannot claim (1) and implement (2). That is how architectural drift starts on day one.

### 4.2 Submodules — pros

1. **Compile-time enforcement.** Domain's `pom.xml` has zero framework dependencies. Spring, Jackson, JPA, Kafka clients don't exist on its classpath. `@Autowired` is not a resolvable symbol.
2. **Framework-free tests.** Domain and application tests run as pure JUnit + Mockito — no Spring, no Testcontainers. Milliseconds, not seconds. Hours of CI time saved per year.
3. **Anaemic domain becomes hard to write.** When `@Autowired` is unavailable in the domain module, developers naturally put logic on the aggregate. The compiler enforces what code review can only suggest. (See §3.6.)
4. **The architecture is the code.** Module tree = architecture diagram. New joiners see the hexagon by reading the file system.
5. **Survives team churn.** Contractors rotate; reviewers change; the compiler does not. Mercury must run for 10+ years.
6. **Replacing Cameron is a two-file change.** Cameron-specific code lives only in `CameronAsaInboundAdapter` and `CameronAsaOutboundAdapter`. Swap connectors, change those two files, done. Under packages-only, Cameron leaks wherever a developer found it convenient.
7. **Refactoring discipline.** Moving a class across the hexagon requires intention. In an OMS where the domain is the crown jewel, friction at that boundary is a feature.
8. **It is actually hexagonal** — see the acid test.

### 4.3 Submodules — cons

1. Slightly more files to touch when renaming a service. IDE handles it.
2. Slightly more decision-making when placing a class. This is a feature: those decisions need to be made anyway.
3. Cross-cutting changes touch more modules. Rare once the domain stabilizes.

### 4.4 Packages-only — pros

1. Fewer `pom.xml` files.
2. Faster initial scaffolding.
3. Easier class relocation during early prototyping.
4. Simpler for developers unfamiliar with multi-module Maven.

### 4.5 Packages-only — cons

1. **Not actually hexagonal.** Structurally identical to standard Spring Boot.
2. **No compile-time enforcement.** First `@Autowired` in a domain class breaks the boundary. It does not heal.
3. **Anaemic domain is the default outcome.** Without friction, domain logic drifts into use cases.
4. **ArchUnit is partial substitute.** Test-time enforcement, violations still compile, rules can be disabled, framework jars remain on the classpath.
5. **Heavier unit tests.** Framework on every test's classpath whether used or not.
6. **Boundary erosion is invisible.** No metric surfaces gradual coupling.

### 4.6 Head-to-head

| Criterion | Submodules | Packages |
|---|---|---|
| Actually implements hexagonal | **Yes** | No (renamed Spring Boot) |
| Enforcement | Compiler | Convention + ArchUnit (optional) |
| Violation detection | Compile time | Test time, or never |
| Framework leak into domain | Impossible | Possible |
| Anaemic domain risk | Low | High (default) |
| Long-term rot resistance | Strong | Weak |
| Fit for a 10-year system | Yes | Degrades |

> **Packages give you a convention. Submodules give you an invariant.**

**Recommendation: submodules.** One-time setup cost. Compounds for a decade.

## 5. Decision 2 — Maven vs Gradle

### 5.1 CPP precedent

Not a greenfield choice in isolation. Mercury sits inside CPP's broader trading/post-trade ecosystem:

| System | Build tool |
|---|---|
| Cosmos (newer order entry) | Gradle |
| Older trading systems | Maven |
| **Post-trade financing (newer rebuild)** | **Maven** |
| Mercury (this proposal) | **Maven** |

Cosmos is the exception, not the precedent. Every other major CPP trading and post-trade system — including the newer post-trade financing rebuild, which is the closest match to Mercury in scope and profile — is Maven. Engineers rotating across CPP trading systems expect Maven. Hiring for Maven is easier. Shared CI conventions across CPP trading are Maven-shaped.

### 5.2 Maven — pros

1. **Native IntelliJ integration.** Maven runs in-process in the IDE; errors surface immediately. Gradle delegates to its daemon (separate JVM); errors surface more slowly, daemon warm-up is a noticeable cost, debugging is harder because the IDE isn't running the build.
2. **Declarative XML is self-describing.** Dependencies, plugins, phases all in fixed locations. No scripting needed to understand a build.
3. **Convention over configuration.** Universal directory layout. Any Java developer can open a Maven project and know where things are.
4. **Reproducibility.** Deterministic builds. No "different on Tuesdays."
5. **Right-sized for our scope.** ~20 standard Java modules. Maven fits; Gradle's flexibility is unused weight.
6. **Lower cognitive load across a 50+ person team.** One way to declare a test dependency. Gradle has three (Groovy DSL, Kotlin DSL, imperative tasks).
7. **Stable across CI, local, and IDE.** What runs in IntelliJ runs in Jenkins.
8. **Alignment with CPP's trading-system pattern.**

### 5.3 Maven — cons

- XML is verbose. Verbosity is the price of declarativeness.
- Marginally slower than Gradle for very large builds. Gap is narrow at 20 modules.

### 5.4 Why Gradle's speed advantage doesn't apply to us

The "Gradle is faster" claim is the most-cited Gradle pro. It deserves examination because in our actual workflow it fades to near zero.

- **CI does full builds.** Pipelines run from clean checkout. Incremental compilation does not apply to cold builds.
- **In the IDE, the IDE handles incremental compilation itself.** When a developer edits a file in IntelliJ, IntelliJ's own compiler recompiles — neither Maven nor Gradle is involved in the edit-compile loop. Build tools only run on explicit build/test/package. With Maven that runs in-process; with Gradle it goes through the daemon.
- **Build caching helps at scale we don't have.** Remote build caches shine in orgs with hundreds of engineers sharing a trusted cache. We are not that.
- **Gradle's real wins** — polyglot repos with 1000+ modules, Android, native compilation — are not our scenarios.

**Felt developer experience is what matters.** Developers feel speed in the IDE, not CI logs. Maven + IntelliJ is at least as fast as Gradle + IntelliJ on our workload, and often faster because error surfacing is more immediate.

### 5.5 Gradle — cons (relevant to us)

1. **Scripts are programs.** Over 10 years, accumulated conditional logic is hard to audit.
2. **Two competing DSLs** (Groovy, Kotlin) — teams split, migration is painful.
3. **IntelliJ integration via daemon** — slower feedback, harder debugging.
4. **Steeper learning curve** — "how do I add a dependency" has multiple correct answers.
5. **Version upgrades break builds** more often than Maven version upgrades.
6. **Style drift** across a large team.
7. **Diverges from CPP's dominant Maven pattern.**

**Recommendation: Maven.** No need for Gradle's flexibility. Strong need for Maven's transparency and IntelliJ integration. Clear alignment with CPP trading conventions.

## 6. Decision 3 — Naming the boot module

The boot module contains the Spring Boot main class, `@Configuration` wiring, `application.yml`, profiles, and health checks. It contains no business logic. Its job is to assemble the other three modules into a deployable artifact.

The question: what do we call it? Three serious candidates: `*-service`, `*-bootstrap`, `*-boot`.

### 6.1 `*-service` (recommended)

**Pros**
1. **Matches the deployable unit.** This module is what gets packaged, deployed, and run as a service. The name describes what it produces.
2. **Aligns with how the team and the business already speak.** "The execution service" is what people say in standups, in Jira, on dashboards. The module name matches the conversational name.
3. **Short and natural.** `execution-service` reads cleanly.
4. **Ops-team intuitive.** When SRE looks at "what's running in prod" the names match.

**Cons**
1. **Term collision with "service" inside hex.** Inside the hexagon, "service" already means application service or domain service. So `execution-service` (the module) contains classes like `RouteOrderUseCase` (an application service) which can be confusing in conversation: *"is the service in the service module?"*
2. **Word is overloaded in Java generally.** "Service" is one of the most overused words in enterprise Java — Spring's `@Service`, Java's `ServiceLoader`, microservice, application service.

### 6.2 `*-bootstrap`

**Pros**
1. **Precise.** The module's job is bootstrapping — wiring up the other modules and starting the runtime.
2. **No collision** with any other concept in the codebase or the broader Spring/Java vocabulary.
3. **Reads as technical infrastructure.** Anyone seeing `execution-bootstrap` knows immediately it's plumbing, not business code.

**Cons**
1. **Less natural in conversation.** People talk about "the execution service," not "the execution bootstrap."
2. **Name describes the *act* (bootstrapping) more than the *artifact* (the deployable).**
3. **Slightly clinical** — works but lacks the everyday clarity of "service."

### 6.3 `*-boot`

**Pros**
1. **Even shorter** than `*-bootstrap`.
2. **Echoes Spring Boot** which is exactly what the module wraps.
3. **No collision** with hex vocabulary.

**Cons**
1. **Tied to one framework** — `*-boot` reads as "this is the Spring Boot module." Fine today, but if we ever changed runtime (Quarkus, Micronaut), the name lies.
2. **Can read informal** — "the boot module" sounds offhand; "the service" sounds substantive.

### 6.4 Why I prefer `*-service`

The concern with `*-service` is real but small and well-managed by documentation and convention. The mitigations:

- Reserve **"service" (the module)** for the deployable. Reserve **"use case"** for the application-layer operation classes (`RouteOrderUseCase`, not `RouteOrderService`). Reserve **"domain service"** when the rare cross-aggregate domain logic needs a class. Three distinct nouns; collision avoided.
- The match between module name and conversational/operational name (Jira, dashboards, ops, standups) is a real win. Naming should serve daily readability, not theoretical orthogonality.

The fallback. If the team feels the collision is too risky, `*-bootstrap` is a strong second choice with no naming drawbacks beyond being slightly less natural in conversation. Either is defensible. I do not recommend `*-app` (collides with `*-application` — that one is genuinely confusing) or `*-runtime` (jargon).

### 6.5 Quick comparison

| | `*-service` | `*-bootstrap` | `*-boot` |
|---|---|---|---|
| Matches deployable artifact | **Strong** | Medium | Medium |
| Matches conversational name | **Strong** | Weak | Weak |
| No collision with hex vocabulary | Weak | **Strong** | **Strong** |
| Framework-neutral | **Strong** | **Strong** | Weak |
| Short and natural | **Strong** | Medium | **Strong** |
| Recommendation | **Pick** | Strong fallback | Acceptable |

**Recommendation: `*-service`**, with `*-bootstrap` as a defensible alternative if the team prefers it.

## 7. Resulting module structure

```
mercury/                                  ← monorepo root (parent pom)
├── pom.xml                               ← parent, dependencyManagement, plugin pinning
│
├── shared/
│   ├── shared-domain-kernel/             ← pure Java: Money, Isin, OrderId, enums
│   ├── shared-events/                    ← Avro/Protobuf schemas + generated code
│   ├── shared-state-machine/             ← generic state machine primitives
│   └── shared-adapter-kit/               ← common Kafka/socket/REST plumbing
│
├── oms-core/
│   ├── oms-core-domain/
│   ├── oms-core-application/
│   ├── oms-core-adapter/
│   └── oms-core-service/
│
├── execution/
│   ├── execution-domain/
│   ├── execution-application/
│   ├── execution-adapter/
│   └── execution-service/
│
├── ticketing/
│   ├── ticketing-domain/
│   ├── ticketing-application/
│   ├── ticketing-adapter/
│   └── ticketing-service/
│
└── bff/
    ├── bff-domain/
    ├── bff-application/
    ├── bff-adapter/
    └── bff-service/
```

### 7.1 Dependency rules (enforced by pom)

| Module | May depend on |
|---|---|
| `*-domain` | `shared-domain-kernel` only. No framework. |
| `*-application` | Its own `*-domain`, `shared-domain-kernel`, `shared-state-machine`. Defines ports. |
| `*-adapter` | Its own `*-application` and `*-domain`, `shared-events`, `shared-adapter-kit`. Imports frameworks. |
| `*-service` | All three siblings. Spring Boot main, configuration, wiring. No business logic. |
| `shared-domain-kernel` | JDK only. |
| `shared-events` | Schema-generator output only. |
| `shared-state-machine` | JDK only. |
| `shared-adapter-kit` | Framework allowed. |

### 7.2 Notes on `shared/` and the BFF

**`shared/` governance.** A class belongs in `shared/` only if all are true: used by ≥2 services today, low change frequency, not service-specific logic in disguise, respects its module's hex layer, safe to change for all consumers together. Duplication is preferred over premature sharing. Architect sign-off on new shared classes. Quarterly pruning.

**BFF.** Same monorepo, top-level module, same hex structure as other services, deployed independently. Atomic cross-cutting changes (BFF + oms-core schema change in one PR), single source of truth for `shared-events`, single CI pipeline. Separating source while contracts remain coupled just hides the coupling.

### 7.3 Cross-service communication

Services communicate via published events (`shared-events`) and REST/gRPC. **Never** by depending on each other's modules. There is no `oms-core-domain` on `execution-adapter`'s classpath.

## 8. Objections and responses

**"Maven submodules are complex."**
False at our scope. Four modules per service is boilerplate. Codebase complexity comes from *dependencies*, not pom count. Hiding dependencies inside one module obscures them, doesn't simplify them. Spring, Spring Boot, Jackson, Hibernate are all multi-module — if it were genuinely complex, the ecosystem would have abandoned it.

**"Packages with good naming are already hexagonal."**
The most important objection because it's often believed sincerely. Every well-organised Spring Boot app has separate packages for controllers, services, repositories. Renaming them to `adapter/inbound`, `application`, `adapter/outbound` changes nothing structurally — same classpath, same `@Autowired` availability, same framework imports possible anywhere. By that logic every well-organised Spring Boot app is hexagonal, which is obviously wrong. The acid test settles it: under submodules the domain compiles without frameworks; under packages-only it cannot.

**"Packages + ArchUnit is enough."**
ArchUnit is defense-in-depth, not a substitute. Test-time enforcement, violations still compile, rules can be disabled, framework remains on the classpath. Use it *in addition to* submodules for intra-module rules — not instead.

**"Some logic in domain, some in application — how do we decide?"**
Section 3 has the full answer with five lenses and a worked example. Short version: domain holds rules intrinsic to the financial concepts (invariants, state transitions, value-object semantics, pure calculations). Application holds orchestration (port calls, transactions, idempotency, sequencing across aggregates). When in doubt, ask whether a regulator could change the rule next quarter without changing what an order *is* — if yes, it's policy and belongs in application.

**"Gradle is faster."**
In theory, not in our workflow. CI does full builds (no incremental benefit). IntelliJ handles edit-compile incrementally itself, not the build tool. Caching helps at scales we don't have. Maven's in-process IntelliJ integration surfaces errors faster than Gradle's daemon-mediated one.

**"Cosmos uses Gradle, so we should too."**
Cosmos is one team's choice in a narrower domain. The post-trade financing rebuild — newer, closer match to Mercury's profile — chose Maven. The closer reference point goes the other way.

**"XML is ugly."**
Aesthetics are not a decision criterion. Declarative XML is unambiguous, diffable, and readable without learning a DSL.

**"Why call the boot module `service`? Won't that clash with application services?"**
Yes, mildly — see Section 6 for the full discussion. The collision is documentable; the upside is that the module name matches the deployable artifact name and the conversational name (the way ops, Jira, and standups already refer to it). Fallback: `*-bootstrap`.

**"Hexagonal is overkill."**
Execution alone has five distinct use cases (route, cancel, replace, process electronic execution reports, record manual voice-trade fills) driven from multiple inbound channels. Some converge on the same domain operation (both fill use cases call `Execution.applyFill(...)`); others operate on different parts of the state machine. That is exactly the scenario hex is designed for. Adding hex after the fact is strictly more expensive than having it from day one.

**"What if we need Gradle's flexibility later?"**
Migrate if a real need emerges. Cost of migration later is smaller than cost of paying for unused flexibility now.

---

## 9. Decision ask

1. **Hexagonal architecture**, enforced structurally — not labeled.
2. **Four Maven submodules per service** — `domain`, `application`, `adapter`, `service`.
3. **Rich domain, thin application.** Domain holds invariants and state machines; application orchestrates via ports. Section 3 sets the framing.
4. **Maven** as the build tool.
5. **Boot module named `*-service`**, with `*-bootstrap` as the fallback if the team strongly prefers no naming collision.
6. **`shared/` hierarchy** with four tightly-scoped modules and governance rules.
7. **No inter-service module dependencies.** Services talk via events and APIs only.
8. **ArchUnit** as defense-in-depth for intra-module rules only.
